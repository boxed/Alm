alm compute · JavaScript vs WebAssembly-GC vs native

The js-framework-benchmark is DOM-bound — every operation crosses into the browser, which hides how fast the compute backend actually is. These are pure-computation workloads (no DOM, no I/O) running the same Main.elm through three alm backends: compiled to JavaScript, to WebAssembly-GC, and to a native binary. Bars show wasm-gc's speed relative to the JS backend; right of the line is faster, left is slower. Lower milliseconds is better.

Where wasm-gc wins

Allocation-heavy work and tight recursion. Lists, dictionaries and sorting run 1.5–2× faster — the compact heap layout and engine GC beat JS object churn. And with unboxed number representation, naive recursion (fib) is now ~2× faster too, where it used to trail.

The last gap

Float-heavy loops. Mandelbrot sits at ~0.9×. Its hot recursive parameters are now unboxed f64, but float values bound in a let are still boxed — closing that would take the last workload past v8 too.

How it was run. Each workload is a value program (main : Int) exposing bench : Int -> Int, compiled by alm to three targets from identical source. JS: the exported bench(size) called in-process under Node/v8. wasm-gc: main_int() on the instantiated module. native: the unboxed --target=native-typed binary (the AOT ceiling). All three are checked to return the same result before timing. Time is the median of 25 timed calls after 10 warmup calls (native: min of 12 process runs, so it includes ~1 ms of process startup). Elm's Int is a 64-bit integer on the wasm/native backends and an IEEE double on JS; workloads keep intermediate values under 253 so all three agree exactly.

Reading it. The native column is a reference ceiling, not a like-for-like: it wins the scalar loops outright but loses the allocation-churn workloads (binary-trees, Dict) to v8's generational GC, where its conservative collector pays more for short-lived garbage.
