The benchmark above is DOM-bound — every operation crosses into the browser, which hides the compute backend's real speed. These are pure-computation workloads (no DOM, no I/O) running the same Elm source through the official elm compiler and alm's three backends — JavaScript, WebAssembly-GC, and a native binary. Against official elm the story splits cleanly: on scalar code (recursion, float math, sorting) alm-js is neck-and-neck — it emits the same shape of JavaScript — but on collections it pulls away, because alm represents List as a growable vector (not elm's linked-list cons cells) and fuses map/filter/fold into specialized per-call loops: the list operations run 2–8× faster than elm on the same JS, and Dict ~1.1×. WasmGC then pulls further ahead — up to ~6.5× the alm-js speed on the list ops — because it stores List Int and List Float as contiguous unboxed i64/f64 arrays (no per-element heap box), and the native backend — its numeric loops auto-vectorizing to SIMD — reaches 50×+ over official elm on list work and 2–3× on recursion/sorting. The three list ops are shown separately because they behave differently: map and filter allocate a fresh unboxed vector (WasmGC builds it in one pass, no per-element node — map now runs at 1.35× native), while fold is a pure traversal whose cost here is the per-element modBy. The float-list row is the same story: a List Float is a raw f64 array on both native and WasmGC, so wasm-gc runs it ~8× faster than its own JS backend and ~50× faster than official elm's boxed linked list.

Compile time

milliseconds · full project · lower is better

How fast is the compiler itself. Each row is a whole project (its own modules plus every dependency) built from a cold cache. alm recompiles everything on every invocation — it has no incremental cache — yet a full alm build lands in 60–130 ms, roughly the time the official elm compiler takes for an incremental rebuild once its dependencies are already cached (elm (incr.)). Measured like-for-like — both doing a full compile from a cleared cache — alm is 5–11× faster than elm (elm (full)), which spends most of its ~680 ms recompiling the dependency set. alm to WebAssembly-GC costs a little more than to JS (extra codegen), and the native path pays LLVM's optimizer (its default -O2; a --dev build skips it and lands in a few seconds). Types are interned (structurally shared via Rc), so the monomorphizer clones them in O(1) rather than deep-copying — part of what keeps these times flat as programs grow.

How the DOM benchmark was run (rebuilt & re-measured 2026-07-21). A keyed js-framework-benchmark app — create/append 1k & 10k rows, update every 10th, swap, select, remove, clear — implemented five ways: elm (official compiler), alm→JS and alm→WebAssembly-GC (both from the same Main.elm; the wasm module runs on a hand-written DOM-host JS shim, no WASI), React 18 (keyed, createRoot) and Svelte 4 (keyed #each). puppeteer-core drives system Google Chrome headless in real time (no virtual clock). Each operation is paint-inclusive: timed from the click to a task scheduled after the next frame's paint (requestAnimationFrame → setTimeout(0)), so it counts the framework's work and the browser's layout/paint. Reported value = median of 3 page reloads, each the median of 15 timed iterations. This measure is fair to frameworks that batch on rAF (elm) and those that commit synchronously (alm/React) alike — both paint on the same frame — which is why sub-frame incremental ops (select, swap, remove) converge near one frame and don't separate the field; the bulk ops (create, append), whose work exceeds a frame, do. Compile/memory/bundle-size figures below are from the earlier full-app run and were not re-measured this pass.

vdom diffing — confirmed in this run. The alm-wasm patch now performs full patch-time diffing: attributes and inline styles are diffed rather than blindly re-applied, dropped keys are removed, boolean DOM properties (disabled, checked, …) are applied and cleared, and event listeners are added, updated in place, and removed — matching the JS backend. Every operation above rendered correctly on the real WebAssembly-GC engine + real DOM, and a separate in-browser check confirmed a handler's captured value updating on patch, a conditionally-added handler firing, and a removed handler going silent. The change is perf-neutral for these operations — an unchanged handler is structurally equal and skipped, so no listener is re-registered on a steady update.

How the compile benchmark was run. Each program is a real package's project (its Main.elm plus every dependency). Time is the min of repeated make runs (min filters scheduler/disk noise; wall clock). alm-js/alm-wasm/alm-native are alm make … --target=js|wasm-gc|native-typed — alm keeps no incremental cache, so each is a full compile of the whole project. elm (full) clears elm-stuff/ before every run so the official compiler recompiles the entire dependency set (an apples-to-apples full build); elm (incr.) keeps that cache, i.e. elm's fast path where only changed modules recompile. Same machine, same source; alm's release binary.

How the computation benchmark was run. Each workload is a value program (main : Int) built from identical source by the official elm 0.19.1 compiler and by alm's three targets. elm: since official elm exposes only Elm.X.init, the workload's bench is driven through a Platform.worker port — elm runs the compute synchronously inside the port send, so that call is timed (result verified on the next tick). alm-js: alm's exported function called in-process under Node/v8. alm-wasm: the module's entry called on the instantiated module. alm-native: the unboxed --target=native-typed binary. All three are checked to return the same result before timing; time is the median of 25 timed calls after 10 warmup. The JS and wasm figures are timed warm and in-process; the native figure is the min of 12 process runs minus a measured startup floor — a no-work binary (dyld + heap init + worker-thread spawn, ~2 ms) — so all three columns are compute-only and directly comparable rather than penalizing native for cold process startup the warm in-process figures never pay. Native uses a custom Immix-lite garbage collector (bump allocation + conservative mark-region reclaim), now the default — it replaced Boehm entirely, roughly halving the allocation-heavy workloads. On this compute-only basis native leads most rows; the two places it doesn't are the weight-balanced Dict tree's pointer-chasing (versus v8's JIT + generational GC) and mandelbrot, where TurboFan's scheduling of the tight scalar-float loop edges LLVM by ~20% even though the emitted loop is already optimal.
